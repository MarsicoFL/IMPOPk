impopₖ — Tutorial

This tutorial walks through the complete impopₖ pipeline using the precomputed example datasets shipped in data/examples/. It covers the conceptual background, the four inference modes, and how to move from the bundled examples to your own pangenome data.

Estimated time: ~1 hour total (15 min reading background, 30 min running the four examples and interpreting outputs, 15 min configuring a run on your own data). You need a laptop with Rust ≥ 1.70 installed; no GPU, no pangenome alignment, and no internet access are required for the first four sections.

1. Background: what is impopₖ doing?

The biological signal

When two individuals share a recent common ancestor, the DNA they co-inherit remains nearly identical in sequence along the chromosomal interval that survived recombination. The more distant the ancestor, the more mutations and structural changes have accumulated, and the lower the per-base identity. This relationship is the foundation of both identity-by-descent (IBD) detection and local ancestry inference (LAI): the length of a high-identity stretch reflects how recent the shared ancestor is, and its chromosomal position locates where that shared ancestry lives in the genome (Kingman, 1982; Thompson, 2013).

Why existing tools require VCFs

Widely used tools — hap-ibd, RFMix, FLARE — operate on phased VCF genotypes. This introduces three costs: SNP ascertainment bias (only catalogued variants are used), loss of structural variation (SVs are dropped), and reliance on statistical phasing (whose errors concentrate at ancestry switch points).

The pangenome alternative

With HPRC v2 and similar projects, hundreds of haplotype-resolved assemblies exist. When they are aligned into a pangenome (e.g. with PGGB) or against a linear reference with impg, the fraction of matching bases per genomic window is a direct, variant-free readout of shared ancestry. impopₖ reads that signal and runs Hidden Markov Models on top of it to decode IBD, local ancestry, founder haplotypes, and kinship — all within a single framework, changing only the number of hidden states.

What you gain. No phasing step, no reference panel of variants, no ascertainment bias, and structural variation is implicitly carried in the alignment identity signal.

What you lose. Very fine per-base resolution at SNP density (impopₖ windows are typically 10 kb), and compute scales quadratically in the number of haplotypes.

2. Installation and first build

What you need

  • Rust toolchain ≥ 1.70 — from rustup.rs.
  • Python 3.8+ — used only by the kinship post-processor and the synthetic data generators.

For the full pipeline starting from a pangenome (Section 9) you also need impg ≥ 0.3 and AGC ≥ 3.2. These are not required for this tutorial — we use precomputed identity tables.

Clone and build

git clone https://github.com/MarsicoFL/IMPOPk.git
cd IMPOPk
cargo build --release

Build should finish in ~40–60 seconds. Confirm the five expected binaries:

ls target/release/ | grep -E '^(ibs|ibd|ibd-validate|ancestry|jacquard)$'
# Expected output:
# ancestry
# ibd
# ibd-validate
# ibs
# jacquard

3. The shared data pipeline

Every inference mode consumes the same primary input — a windowed pairwise identity TSV — and adds its own downstream HMM or post-processor on top: 

   Pangenome alignment (PAF) + assemblies (AGC)
                 │
                 │    ibs  (wraps impg similarity)
                 ▼
   Windowed pairwise identity TSV   ◄──── input/ibs.tsv in each example
                 │
   ┌─────────────┼────────────────┬─────────────────┐
   ▼             ▼                ▼                 ▼
 (awk)          ibd            ancestry         jacquard (Δ)
  IBS         IBD segs       painted tracts       9 coefs
  scan                            │
                                  ▼
                     scripts/kinship_from_ibd.py
                           kinship θ̂

The ibs.tsv format is documented in the top-level README; briefly:

chrom    start    end    group.a    group.b    estimated.identity    [a.length]    [b.length]

All five precomputed example folders under data/examples/ ship such a file. The format is the same whether ibs produced it from a real pangenome or one of the synthetic generators in scripts/ did.

4. Mode 1 — IBS window enrichment

Concept

IBS (identity-by-state) is the simplest mode: for each window, count how many haplotype pairs within a population share identity above a threshold (default 0.9999). Windows of elevated within-population sharing indicate either true IBD clusters (e.g. nearly-fixed haplotypes under positive selection) or demographic bottlenecks. The paper's chr2 LCT selection scan uses exactly this quantity.

Given a pairwise identity row (a, b, w) with identity S(a, b; w), the IBS-enrichment statistic in window w for a population P is:

E(w; P) = |{ (a,b) ∈ P×P : S(a,b;w) ≥ θ }| / |{ (a,b) ∈ P×P }|

Run the example

cd data/examples/ibs
bash run.sh

Input: input/ibs.tsv contains 100 windows of a 1 Mb slice of chr12 for ~15 HPRC EUR haplotypes. The script has no HMM; it is a single awk pass that computes the IBS fraction per window.

Interpret the output

cat output/ibs_enrichment.tsv | head -5
# chrom          start     end        n_pairs  n_ibs  frac_ibs
# CHM13#0#chr12  15000000  15009999   15       1      0.066667
# CHM13#0#chr12  15010000  15019999   15       3      0.200000
# CHM13#0#chr12  15020000  15029999   15       3      0.200000
# CHM13#0#chr12  15030000  15039999   15       3      0.200000

Sort by frac_ibs descending to see the windows where the EUR panel is most shared. In a real selection scan this would be followed by a permutation-based null (see paper §Results), but the core statistic is already computed here.

5. Mode 2 — IBD detection (2-state HMM)

Concept

For each ordered pair of haplotypes, the ibd binary fits a 2-state Hidden Markov Model (IBD vs non-IBD) with Gaussian emissions. The signal the model sees per window is the raw pairwise identity S(a, b; w):

  • Emissions — two Gaussians, one per state. Means and variances are initialised from k-means on the observed identities, then refined by Baum-Welch EM.
  • Transitions — a small prior probability of entering IBD (--p-enter-ibd, default 1e-4) and a segment-length prior that scales with the chromosome.
  • Decode — Viterbi gives the maximum-likelihood state path, and a forward-backward pass gives per-window posteriors. Segments shorter than --min-len-bp (default 2 Mb) are discarded.

Run the example

cd data/examples/ibd
bash run.sh

The example input is synthetic — produced by scripts/generate_synthetic_ibd_example.py — because public HPRC haplotypes are unrelated and would yield zero true IBD. The synthetic data contains two known IBD segments:

  • SIM1#1 × SIM2#1: 1.25 Mb – 3.75 Mb (2.5 Mb)
  • SIM1#2 × SIM3#1: 0.50 Mb – 2.00 Mb (1.5 Mb)

Interpret the output

cat output/ibd.tsv
# chrom  start    end      group.a  group.b  n_windows  mean_identity  mean_posterior  min_posterior  max_posterior  lod_score
# chr12  1250000  3750000  SIM1#1   SIM2#1   250        0.999941       1.0000          1.0000          1.0000         2280.21
# chr12   500000  2000000  SIM1#2   SIM3#1   150        0.999948       1.0000          1.0000          1.0000         1446.78

Both ground-truth segments are recovered exactly, with posterior 1.0 at every window and LOD scores in the thousands (log-odds of IBD vs non-IBD). mean_identity ~0.9999 matches what the generator wrote; mean_posterior is the average of per-window posteriors across the segment.

Running on real data. If you swap input/ibs.tsv for a real pangenome-derived TSV (Section 9), also update --region, --region-length, and pick a population prior matching the panel (AFR, EUR, EAS, CSA, AMR, or Generic). See ibd --help.

6. Mode 3 — Local ancestry (N-state HMM)

Concept

For a query haplotype q and K reference populations with labelled haplotypes, the ancestry binary decodes each 10 kb window into a state k ∈ {1, …, K}.

The per-window log-emission for state k is:

log P(o_w | s_w = k) = f_k(w) / τ − log Σj exp(f_j(w) / τ)

where fk(w) = max S(q, h; w) over haplotypes h in population k. τ is a softmax temperature chosen adaptively per pair of populations via a Bradley-Terry pairwise-contrast model: closely-related populations get a sharper τ than well-separated ones. Transitions are governed by a small switch probability (--switch-prob).

Run the example

cd data/examples/ancestry
bash run.sh

Inputs:

  • input/ibs.tsv — windowed identities across a 50 Mb stretch of chr12 for a panel of AFR + EUR + AMR HPRC haplotypes.
  • input/populations.tsv — mapping haplotype → population.
  • input/queries.txt — the haplotypes to paint.

Interpret the output

head -5 output/ancestry.tsv
# chrom  start     end        sample     ancestry  n_windows  mean_similarity  mean_posterior  discriminability  lod_score
# chr12  15000000  15329999   HG00639#2  EUR       33         0.999906         0.999207        0.000667          2.1735
# chr12  15000000  15329999   HG00639#1  EUR       33         0.999915         0.601676        0.000285          0.0530
# ...

Columns you should care about:

  • ancestry — the decoded state for that tract.
  • mean_posterior — Viterbi-path posterior average. Values below ~0.7 indicate that the HMM could not confidently separate states across the tract.
  • discriminability — the per-window gap between the top-two state probabilities. High = clear call. Low (e.g. 0.0003) = the window could plausibly belong to more than one population.
  • lod_score — log-odds of the assigned state vs the second-best alternative.
Small-panel caveat. This bundled example uses only a 46-hap panel on a 50 Mb slice, so absolute posteriors are modest. The paper's validation uses 10 chimeras × 133 Mb chr12 and reaches 95–97% concordance with RFMix / FLARE; see §Results in the manuscript.

7. Mode 4 — Founder painting (pedigree)

Concept

The same ancestry binary is used — but now each state is an individual founder haplotype, not a population. For an N-grandparent pedigree you run the N-state HMM with N = number of founders. The paper validates this on the CEPH 1463 Platinum Pedigree (4 grandparents → 4 states) with 99.2% concordance against ground-truth inheritance vectors across all 22 autosomes.

Run the example

cd data/examples/pedigree
bash run.sh

The example input is synthetic (generated by scripts/generate_synthetic_pedigree_example.py) and describes a descendant CHILD that is a four-block mosaic of founders GP1–GP4:

RegionTrue founder
0 – 1.25 MbGP1
1.25 – 2.50 MbGP2
2.50 – 3.75 MbGP3
3.75 – 5.00 MbGP4

Interpret the output

cat output/painting.tsv
# chrom              start     end       sample  ancestry  n_windows  mean_similarity  ...
# CHM13#0#chr12      0         1250000   CHILD   GP1       125        0.999939         ...
# CHM13#0#chr12      1250000   2500000   CHILD   GP2       125        0.999944         ...
# CHM13#0#chr12      2500000   3750000   CHILD   GP3       125        0.999947         ...
# CHM13#0#chr12      3750000   5000000   CHILD   GP4       125        0.999947         ...

All four blocks are recovered at the exact ground-truth boundaries. discriminability is high and the posterior is > 0.99. The same configuration, applied to a real pedigree whose founders are sequenced as four independent assemblies, produces the CEPH 1463 founder painting in the paper.

8. Mode 5 — Kinship (scalar θ and Jacquard Δ)

impopₖ reports kinship in two complementary forms.

8a. Scalar kinship θ̂ from detected IBD

The paper's formula is a straightforward summation over the four haplotype pairings of a diploid pair:

θ̂ = Σα,β ∈ {1,2} LIBD(Aα, Bβ) / (4 · L)

This is implemented as a post-processor over the ibd binary's output:

python3 scripts/kinship_from_ibd.py \
  --ibd          data/examples/ibd/expected_output/ibd.tsv \
  --chrom-length 5000000
# individual_a  individual_b  total_ibd_bp  theta_hat
# SIM1          SIM2          2500000       0.125000
# SIM1          SIM3          1500000       0.075000

Full siblings should score θ̂ ≈ 0.25, half-siblings and grandparent relationships ≈ 0.125, first cousins ≈ 0.0625. The synthetic example above recovers θ̂ ≈ 0.125 on a fabricated half-sibling-like pair.

8b. Full Jacquard Δ coefficients

The jacquard binary goes beyond the scalar by returning the nine condensed-identity coefficients Δ1 … Δ9 for a diploid pair. It reads the IBS table (not the IBD segments) and the four haplotype IDs:

target/release/jacquard \
  --ibs    data/examples/ibd/input/ibs.tsv \
  --hap-a1 SIM1#1  --hap-a2 SIM1#2 \
  --hap-b1 SIM2#1  --hap-b2 SIM2#2

The nine deltas are printed to stdout. Use this whenever you need the full identity-state decomposition (e.g. to distinguish inbred and non-inbred relationships of the same θ).

9. Running on your own pangenome

Prerequisites

  • A PAF alignment. For human data the HPRC portal provides HPRCv2 PAF files against CHM13. For other species, align your haplotype-resolved assemblies with minimap2 (-x asm5 for intra-species) or build a full pangenome graph with PGGB.
  • An AGC archive of the assemblies — see refresh-bio/agc.
  • impg installed.

1. Build your subset / population files

panel_subset.txt (one haplotype or sample ID per line):

HG00097#1
HG00097#2
HG01884#1
HG01884#2
NA19017#1
NA19017#2

populations.tsv (tab-separated, two columns):

EUR    HG00097#1
EUR    HG00097#2
AFR    HG01884#1
AFR    HG01884#2
AMR    NA19017#1
AMR    NA19017#2

queries.txt (haplotypes you want to paint):

NA19017#1
NA19017#2

2. Compute windowed pairwise identity

ibs -a                      data/alignments/hprc_chr12.paf.gz \
    --sequence-files        data/assemblies/HPRC_r2.agc \
    -r                      CHM13 \
    --region                chr12:1-133324548 \
    --size                  10000 \
    --subset-sequence-list  panel_subset.txt \
    --threads               8 \
    --output                ibs_chr12.tsv

This runs impg similarity over every 10 kb window in the requested region and filters, canonicalises, and streams its output. Runtime scales roughly linearly with the region length and quadratically with panel size. Reference figures: ~8 min for a 133 Mb chromosome at 46 haplotypes on 8 threads; memory is the dominant cost and is driven by impg's per-window graph query.

3. Run the HMMs on your TSV

# IBD
ibd --similarity-file ibs_chr12.tsv \
    --region chr12:1-133324548 --region-length 133324548 \
    --size 10000 --min-len-bp 2000000 \
    --population Generic \
    --threads 8 --output ibd_chr12.tsv

# Local ancestry
ancestry --similarity-file ibs_chr12.tsv \
         --window-size 10000 \
         --populations  populations.tsv \
         --query-samples queries.txt \
         --emission-model max --estimate-params \
         --threads 8 --output ancestry_chr12.tsv

# Kinship
python3 scripts/kinship_from_ibd.py \
        --ibd ibd_chr12.tsv \
        --chrom-length 133324548

10. Troubleshooting

SymptomLikely causeFix
Build fails with impg linker error System-wide impg version mismatch. impopₖ does not statically depend on impg at build time — the ibs binary calls it at runtime. Make sure which impg returns a ≥ 0.3 binary.
ibd reports 0 segments No true IBD in the panel, or all segments are shorter than --min-len-bp. Lower --min-len-bp to 500 kb for exploratory runs. Confirm the panel actually contains related individuals.
ancestry reports high discriminability but low mean_posterior Panel too small to separate close populations. Expand the reference panel, or merge populations at the continental level instead of sub-continental.
WARNING: >30% of windows are uncertain Identity contrast Δ between populations is near the noise floor. Increase window size (--window-size 25000) or move to a continental-level contrast.
jacquard: "haplotype not found" The haplotype ID in --hap-a1 etc. does not match the IBS TSV group.a/group.b format. Use the short form (HG00097#1). impopₖ strips anything after the second # when matching.
Out of memory on a real pangenome Full HPRCv2 panel (~290 sequences) OOMs at ~49 GB per window with the default impg behaviour. Use a reduced panel of ~40 haplotypes (5 per population × 8 populations) as described in the paper's Methods. Most signals are recoverable at this scale.
